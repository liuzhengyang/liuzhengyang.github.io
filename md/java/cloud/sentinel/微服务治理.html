# 线上发布稳定性

可灰度、可观测、可回滚。

  • 无损伤下线（优雅下线） - 先反注册，注册中心将变化更新到主调方，再关闭服务（关闭时等待一段时间，等待主调服务列表更新，没有调用后下线。如果在反注册后仍有调用，可以在返回结果中增加特殊标记告知主调获取最新列表）
  • 无损上线 - 预热（资源加载、连接创建）、初始化。延迟注册、服务就绪检查、主动预热、小流量预热（缓慢提升实例权重）。
  • 微服务就绪检查 - kubernetes 探针技术，存活探针、就绪探针、启动探针。滚动发布新实例发布和老实例下线的顺序问题。

注册中心更新一致性、及时性。 服务启动初始化速度、服务就绪/健康检查(health check)、注册时机

# 全链路灰度解决方案

  • 标签路由: 自定义标签、版本、AZ、压测、场景、Region
  • 节点打标: k8s, 自定义环境变量
  • 流量染色：header, 扩展字段（类似tracing）
  • 分布式链路追踪: header, 扩展字段（类似tracing）

自动检测诊断

  • 应用变更: 应用发布、配置变更
  • 大请求: 请求响应体大小
  • qps突增: 某个主调业务qps突增
  • 下游问题: 下游服务故障
  • 宿主机: 特定的Host/Ip负载高
  • 服务资深问题: gc导致超时、性能诊断
  • 单机问题: fullgc, 线程池满、OOM、磁盘满、磁盘使用率高、忘了问题
  • 中间问题: redis, mq, mysql
  • 业务代码问题:
  • JVM参数: 参数配置不合理
  • 异常: 某类异常突增

服务启动耗时：类加载数量、耗时，bean加载耗时、数量。jar包依赖分析
