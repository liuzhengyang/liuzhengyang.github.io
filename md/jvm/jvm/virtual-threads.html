# virtual thread

Virtual Thread即虚拟线程是JDK19中提供的Java语音新特性，是很多Java开发者十分期待的功能。

# 介绍

jdk19中将会提供virtual thread也就是虚拟线程（或称为轻量级线程、协程）。

相比很多同学都听说过go语言中的协程，创建使用协程十分轻量。Java这次提出的virtual thread也和go的协程、kotlin中的coroutine一致，是同一个概念。

virtual thread是一种轻量化的thread线程，创建virtual thread的成本基本等同于创建普通的对象。而创建原来的普通线程，则会在操作系统中 创建一个对应的线程，并且要分配线程栈（通过-Xss配置）等资源，并且操作系统还会限制线程数量的最大值。

通过virtual thread，能够极大降低编写维护高并发高吞吐的应用的成本。

为了理解虚拟线程，我们可以把它想象成线程池中的任务，而底层真正的系统线程，是线程池中的worker。 全部虚拟线程的背后会有一定数量的系统线程（ForkJoinPool，但是和stream api的forkjoinpool线程池隔离)， 用户创建虚拟线程并指定对应的Runnable任务后，会提交给ForkJoinPool也就是系统线程去执行， 当虚拟线程的任务遇到阻塞操作（比如sleep、网络调用、AQS相关同步器的acquire加锁等）时，会调用VirtualThread的doPark方法， doPark不会阻塞底层的系统线程，而是会通知底层线程去执行其他的可以执行的虚拟线程，当虚拟线程从阻塞状态恢复会运行态时（比如sleep时间到、网络调用返回、 AQS相关同步器release释放等），就会继续回到系统线程的运行对了中，可以被空闲的系统线程运行。

这样只需要很少的系统线程，就能够支持大量的阻塞类虚拟线程的执行，非常适合IO密集型的业务场景。

在虚拟线程出现之前，Java领域解决IO密集型服务（比如http服务处理db、远程cache、rpc等io操作）的高并发问题，都使用 通过基于线程池的框架，接收到一个请求时，会提交给线程池去执行也就是一个请求一个线程的模型， 服务的并发能力受最大线程的数量限制。 当前线程数量不是越多越好，线程数量非常多会带来内存增大、CPU调度开销增大的问题。

# 一些概念

  • thread: 线程，在jdk19之前，Java中的线程和底层操作系统的线程一一对应
  • virtual thread: 虚拟线程、协程，指运行在底层线程上的更上一层的线程，表示一个可运行的任务。
  • carrier thread: 负责运行virtual thread的线程。

# 目标

  • 能够让thread-per-request形式的应用程序充分利用硬件资源
  • 能让已经使用Thread API的代码，只需要最小的改动就可以使用virtual thread
  • 通过已有的JDK工具就可以非常简单的排查、debug、profile分析virtual thread

# thread-per-request style

server应用通常能够处理并发请求，thread-per-request上最常见的处理模型， 即一个线程处理一个请求，这种模型最利于理解和开发、debug、profile。 这种模型的可扩展性可以通过「排队理论 Little's Law」描述，线程数 = QPS * 请求处理时间。 比如现在接口的QPS是2000，每个接口请求耗时为500ms，则系统需要1000个线程。

这种方式的问题是JDK的线程的实现是对操作系统线程的封装，线程的开销比较大，不能创建 太多线程，比如每个线程都有自己的内存（Java里的Xss参数），线程多了之后上下文切换的开销也会更大。

# AQS类的阻塞支持

Java中的AQS提供了同步器的基础实现，通过AQS实现了Lock, Semaphore, CountDownLatch等上层同步器。 同步器具有线程等待、唤醒功能，使用的是LockSupport类的park和unpark实现的。 所以jdk19在LockSupport类中为virtual thread支持park和unpark，就能实现协程阻塞时，把任务交给其他virtual thread而不阻塞 底层的carrier thread。

例如如下LockSupport的park方法实现，会判断当前线程是否是virtual，如果是，调用的VirtualThreads.park。

public static void park(Object blocker) {
    Thread t = Thread.currentThread();
    setBlocker(t, blocker);
    try {
        if (t.isVirtual()) {
            VirtualThreads.park();
        } else {
            U.park(false, 0L);
        }
    } finally {
        setBlocker(t, null);
    }
}

doPark方法通过调用yieldContinuation方法让出计算资源，让底层的carrier thread能够运行其他virtual thread。 在一些情况下，有可能让出失败，比如是在synchronized中的加锁。频繁yield失败，可能会导致carrier thread阻塞，导致系统卡死。 可以通过-Djdk.tracePinnedThread=full启动参数来查看相关的线程栈找到对应代码。

private void doPark() {
    // complete immediately if parking permit available or interrupted
    if (getAndSetParkPermit(false) || interrupted)
        return;

    // park the thread
    setState(PARKING);
    try {
        if (!yieldContinuation()) {
            // park on the carrier thread when pinned
            parkOnCarrierThread(false, 0);
        }
    } finally {
        assert (Thread.currentThread() == this) && (state() == RUNNING);
    }
}

yieldContinuation方法调用Continuation.yield，yield返回后，finally块中调用mount回重新mount回carrier thread。

private boolean yieldContinuation() {
    boolean notifyJvmti = notifyJvmtiEvents;

    // unmount
    if (notifyJvmti) notifyJvmtiUnmountBegin(false);
    unmount();
    try {
        return Continuation.yield(VTHREAD_SCOPE);
    } finally {
        // re-mount
        mount();
        if (notifyJvmti) notifyJvmtiMountEnd(false);
    }
}

public static boolean yield(ContinuationScope scope) {
        Continuation cont = JLA.getContinuation(currentCarrierThread());
        Continuation c;
        for (c = cont; c != null && c.scope != scope; c = c.parent)
            ;
        if (c == null)
