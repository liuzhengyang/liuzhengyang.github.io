# load_new_class_versions

find_sorted_affected_classes: 找到更新的类和它所有的子类（包括子类的子类等），并按照继承关系进行拓扑排序。 对要redefine的类klass设置_redefine_flags标识(MarkedAsAffected)。 遍历ClassLoaderDataGraph，对所有的类，查看类自身以及所有的父类、接口中是否有MarkedAsAffected标识，如果有，加到_affected_klasses数组中。

对于每个_affected_klass。

  • 查找类的新class_bytes，如果这个类正在被redefine（参数里明确指定的），则返回被redefine的class bytes，否则返回之前类的class bytes。
  • 如果是匿名内部类，从class bytes解析出InstanceKlass，记为k。修改class_loader_data为原来类的class_loader_data
  • 如果不是匿名内部类，从class bytes解析出InstanceKlass，记为k。
  • 设置the_class的new_version为新创建出来的InstanceKlass k。

解析类的resolve_from_stream过程，会创建InstanceKlass，并添加到systemDictionary中替换之前的old_klass。

calculate_redefinition_flags: 计算新旧类的差异。(field change, method change, supertype change等) 如果出现了删除类的supertype，目前是不支持更新的，抛出异常。

如果出现field修改，计算复制旧字段到新字段需要的offset, size等信息，用于后面的gc迁移旧对象字段到新对象位置字段使用。 如果没修改，只会复制旧的类的JVM_ACC_FIELD_MODIFICATION_WATCHED等JVMTI flag到新字段（这些是debug时动态添加的，不在class bytes中）

link、verify，链接验证新类。

# clear_cpool_cache

扫描所有类。 调整常量池对于匿名内部类，修改host_klass为新的host_klass(旧host_klass的newest_version()) 查看如果有对旧的类的常量池引用，修改为引用newst_version。 清理常量池缓存，这样下次使用常量池缓存时会重新解析。

# ChangePointersOopClosure,ChangePointersObjectClosure

ChangePointersObjectClosure

  • 如果旧类字段和新类兼容，只更新mark word里类的reference
  • 否则如果新类的对象大小比旧的小，则重新shuffle字段，并把新旧类的大小gap填上"dead_space"
  • 否则，设置_needs_instance_update标识，在full gc的时候，会整理对象。
