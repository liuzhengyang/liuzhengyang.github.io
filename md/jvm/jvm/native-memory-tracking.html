# Java对外内存分析Native Memory Tracking

# 介绍、使用

Native Memory Tracking是Java提供的来分析native memory的工具，简称NMT，NMT需要在启动程序前增加-XX:NativeMemoryTracking=summary 启动参数，才能使用。

要注意的是增加这个参数会增加一定（约5%到10%）的性能开销，所以开启时要评估好风险。

可以发现每一部分都分成了reserved和committed两个部分，这两个名称的含义是什么呢？

reserved可以理解为进程向操作系统声明我需要使用的内存空间大小，操作系统分配的是虚拟内存，并不会立刻给进程分配真正的物理内存。 committed表示进程真正在使用的内存大小，对应的是物理内存中的空间。 比如通过java -Xms1G -Xmx2G启动程序时，最开始reserved是2G，committed是1G，随着java堆内存增加，committed也会增加。

一般占用内存比较多的几个部分介绍如下

  • Class: metadata和classspace，包含class信息，通过-XX:MaxMetaspaceSize能设置metaspace的总大小,classpace可以通过-XX:CompressedClassSpaceSize设置
  • Thread: 线程栈，大小和ThreadStackSize或Xss配置以及线程数有关
  • GC: GC垃圾回收器自己使用的内存，比如g1中的remembered set, mark stack等
  • Code: JIT编译的机器代码，通过-XX:ReservedCodeCacheSize能限制大小
  • Other: 主要是DirectMemory，比如nio框架使用的DirectBuffer，通过-XX:MaxDirectMemorySize能限制DirectMemory的大小
  • Symbol: 字符串常量和常量池

# 查看增长

有时候我们发现程序的堆外内存不断增长，为了避免超出内存限制（比如机器内存或容器内存限制）而oom，我们需要定位内存增长的原因，需要知道 到底是哪里的内存在增加。

jcmd VM.native_memory提供了diff的功能

首先，通过baseline参数设定一个基准线

# pid换成自己的进程id
jcmd pid VM.native_memory baseline
1
2

然后在一段时间后，通过summary.diff或detail.diff参数来查看内存的变化。

# pid换成自己的进程id
jcmd pid VM.native_memory summary.diff
1
2
