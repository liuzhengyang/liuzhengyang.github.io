# redefine

# 基本实现思路

  • 解析传过来的class byte数组；验证参数（包括后面解析之后的方法验证等）；Parse成Klass，记为scratch_class，使用的class loader是the_class的class loader。
  • 要被redefine的class记为the_class
  • scratch_class中的数据最终会复制到the_class，这样系统中其他位置对于the_class的引用不用修改（比如heap中对象头的metadata指针）。
  • 清理codecache等缓存

执行阶段

  • doit_prologue：在JavaThread中运行，解析构建scratch_class、link the_class(如果没link过，link过程中会初始化vtable itable)、合并constant pool重写字节码
  • doit：在VM Thread中运行，deoptimize compiled code；替换scratch_class中的方法到the_class；调整其他类到the_class的引用(利用ClassLoaderDataGraph::classes_do来遍历所有类和方法等)
  • doit_epilogue：清理doit_prologue和doit中申请的内存

# 执行步骤

RedefineClasses API用来实现修改一个或多个类的定义。 我们按照修改一个类的场景下进行讨论，把当前的类成为 the_class，要修改的新类成为 "scratch_class"， 使用the_class是因为只有一个表示特定类的机构，redefinition不会替换成一个新的class对象，只会替换其中的一部分数据。 使用"scratch_class"（scratch意为草稿的）是因为这个scratch_class用来保存新类的一部分数据，这些数据会进行合并再放到the_class中, redefine结束scratch_class就没有用了。

RedefineClasses执行分为几个步骤，doit_prologue(), doit()和doit_epilogue()

  1. doit_prologue表示前序准备阶段，要在safepoint中由JavaThread运行。会检查参数然后
    • 解析新类的定义
    • 链接(link)scratch_class
    • 合并constant polls常量池，并根据合并的常量池按需修改bytecode
    • 验证scratch_class的bytecode
    • 设置常量池缓存，并且修改bytecode来使用缓存
    • 把scratch_class和the_cass进行比较，验证scratch_class是合法的替换类
    • 如果一切正常，保存scratch_class，在后面的doit中使用
  2. doit()被VMThread调用，在safepoint中运行。
    • 获取scratch_class
    • house keeping(清理breakpoints和cache，deoptimize依赖当前类的JIT编译的代码）
    • 把scratch_class中的数据替换到the_class中
    • 给the_class中被淘汰但是仍然感兴趣的部分增加weakreference
    • 调整其他类的对the_class有方法引用的常量池缓存和vtable。这个调整使用ClassLoaderDataGraph::classes_do()
    • 通知SystemDictionary
  3. doit_epilogue()方法在VM op结束和safepoint完成后被JavaThread调用，负责清理前面操作申请的内存。

常量池的细节：

redefine the_class时，我们不能直接把scratch_class的常量池替换到the_class中，因为可能有旧的方法在运行， 可能导致常量池错误，所以为了兼容旧方法和新方法同时使用，需要对常量池进行合并，并修改新方法的字节码来使用合并后的常量池。

合并时，先把旧的常量池复制到合并常量中，保存之前的顺序。然后把scratch_class中的唯一的常量池entry追加到常量池中， 并且记录常量池索引index的变化。 合并完成后，旧方法不会察觉到任何变化，对于新的bytecode，需要修改使用合并常量池。
