# WispEventPump

IO等待是上下文切换的重要原因。 IO分为两类，一类是作为服务端(server，比如tomcat、grpc server)，接收外部请求， 处理请求，返回响应。 io server的线程模型一般为， 1个线程负责accept，负责处理客户端建连请求，完成channel建立。 N个io线程负责channel的读写。 M个worker线程负责对请求进行处理生成返回结果。

交互流程为，accept线程接收请求建连，创建channel并绑定到io线程， io线程监听有请求数据（有数据可读），读取数据(read)，读取完后把byte数据 解码成请求模型，交给worker线程处理，worker线程处理完成得到结果， 编码成byte数据，交给io线程，io线程在channel可写的时候，写回(write)数据给远程。

当前Java的io普遍使用NIO，NIO中分为多个角色。

server端NIO的使用流程为，创建ServerSocketChannel，创建Selector 调用register把selector和ops（关心的操作事件）注册上。

然后不断调用select，select在有关心的事件发生后（或超过select传入的超时时间）会返回， 返回SelectionKey集合，通过SelectionKey可以得到对应的事件的Channel，并且能够判断具体的事件类型（一个SelectionKey可以包含多个）。 事件分为accept、read、write几种。 accept，对客户端建立连接。 read事件调用read接收请求数据。 write事件把返回结果返回。

# Epoll

epoll: 我们可以把NIO看做是Java对于非阻塞IO的接口抽象，epoll是linux时的非阻塞IO实现，Java在linux时底层会使用epoll作为NIO的接口实现。
